**Document:** Product Vision & Core Requirements

## 1. Executive Summary & Product Vision
Standard AI chat interfaces treat conversations as ephemeral, disposable interactions. Users generate thousands of words of valuable insights, code, and research, only for it to be lost in a noisy, unorganized sidebar. 

**The Vision:** BetterChatGPT bridges the gap between an AI Chatbot and a Personal Knowledge Management (PKM) system. It is designed for power users who need to not just *generate* text, but **extract, organize, branch, and reuse** that knowledge efficiently, all within a highly polished, developer-friendly UI.

## 2. Target Audience & Core Motivations
*   **The Persona:** Developers, researchers, writers, and power users who use LLMs daily for complex problem-solving.
*   **The Pain Points:**
    *   *The "Needle in a Haystack":* Finding a specific 50-word insight inside a 3,000-word chat history is frustrating.
    *   *The "Context Tax":* Manually copy-pasting context from old chats into new prompts is tedious.
    *   *The "Messy Sidebar":* A flat list of 200 chats named "New Chat" becomes unnavigable.
    *   *Poor Formatting:* Standard markdown isn't enough for complex math or architecture diagrams.

## 3. Core Feature Specifications

### 3.1. Knowledge Extraction (Highlights & Notes)
*   **Motivation:** AI output is 80% noise and 20% signal. Users need a way to save the signal without losing the context of how it was generated.
*   **Requirements:**
    *   **Highlights:** Users can select any text within an AI's response and save it as a "Highlight" (visually marked in the UI).
    *   **Scratchpad Notes:** Users can attach markdown-supported notes to a specific chat or a specific highlight (e.g., "Summary," "Todo," "Scratch").
    *   **Secondary Panel:** A dedicated UI panel alongside the chat that aggregates all highlights and notes for the active conversation.

### 3.2. Conversation Branching (Clone & Truncate)
*   **Motivation:** Users need to explore alternate conversation paths or fix a bad prompt without losing their existing, working chat history.
*   **Requirements:**
    *   **Clone Up To Here:** Every message has a "Clone" action button. Clicking it creates a completely new Chat containing all messages from the start up to the selected message.
    *   **Edit & Re-generate:** Users can edit any past message. Submitting the edit triggers a warning: *"This will delete all subsequent messages."* Upon confirmation, the chat is truncated, and the AI generates a new response.

### 3.3. Rich Content Rendering
*   **Motivation:** Power users work with code, math, and system architectures. Plain text is insufficient.
*   **Requirements:**
    *   **Code Syntax Highlighting:** Auto-detection and highlighting for 40+ languages with a "Copy Code" button.
    *   **Mathematics:** LaTeX/KaTeX rendering for inline (`$...$`) and block (`$$...$$`) equations.
    *   **Mermaid Diagrams:** Native rendering of flowcharts, sequence diagrams, and architecture graphs generated by the AI.

### 3.4. Smart Input & UX Quality of Life
*   **Motivation:** The app must feel like a native, keyboard-first power tool.
*   **Requirements:**
    *   **File Drop Extraction:** Users can drag and drop text-based files (`.txt`, `.md`, `.json`, `.js`, etc.) into the composer. The app extracts the text in-memory and appends it to the prompt (no cloud storage required).
    *   **Keyboard Shortcuts:** `Ctrl/Cmd + Enter` to submit, `Shift + Enter` for new lines, `Escape` to close modals/cancel edits.
    *   **Theme Switcher:** Native Light/Dark mode toggle powered by DaisyUI.
    *   **Message Actions:** Every AI message includes "Copy to Clipboard" and "Stop Generation" (abort stream) buttons.

### 3.5. Organization & Context
*   **Requirements:**
    *   **Auto-Title Generation:** After the first message exchange, the system automatically generates a concise, 5-word title for the chat using a fast, cheap model (e.g., `gpt-4o-mini`).
    *   **One-Level Folders & Tags:** Flat folder structure for grouping chats, plus JSONB tags (`#urgent`, `#code`) for cross-cutting organization.
    *   **Context Injection (`@` Mentions):** Typing `@` in the composer opens a popover to search past Chats/Notes. The frontend invisibly prepends the referenced text to the system prompt.
    *   **Semantic Search:** Global search bar querying `pgvector` embeddings to find conceptually similar past messages.

### 3.6. Multi-Model Support
*   **Requirements:**
    *   Users can select the AI provider/model (e.g., Claude 3.5 Sonnet, GPT-4o) on a per-chat basis, powered by the Vercel AI SDK.

## 4. Out of Scope for V1 (The "YAGNI" Parking Lot)
To ensure we ship a fast, stable, commercial-grade product, the following features from legacy specifications are explicitly deferred to V2:
*   **Token Counting & Cost Estimation:** (Requires heavy tokenizer libraries and constant pricing updates).
*   **Advanced LLM Sliders:** (Temperature, Top P, Frequency Penalties are hidden to keep the UI clean).
*   **Cloud File Hosting (S3):** (Attachments are parsed for text in-memory and discarded).
*   **Import/Export (100MB JSONs):** (Deferred to focus on core database stability).
*   **i18n (Multi-language UI):** (English only for V1 to speed up component development).
*   **Bring Your Own Key (BYOK):** (This is a SaaS model; API keys are managed securely on the server, not in the user's `localStorage`).

## 4. Key User Workflows

**Workflow A: The Safe Refactor (Using Clone)**
1. User has a 20-message chat where the AI successfully wrote a Python script.
2. User wants to see if the AI can rewrite it in Rust, but doesn't want to ruin the Python context.
3. User clicks "Clone up to here" on the last good message.
4. A new chat opens. User types: *"Actually, rewrite all of that in Rust."*
5. The original Python chat remains perfectly intact.

## 5. Out of Scope for V1 (The "YAGNI" Parking Lot)
*   **Nested Folders:** (Restricted to one-level to keep drag-and-drop state management simple).
*   **Inline Message Trees:** (Replaced by the simpler "Clone & Truncate" feature).
*   **Cloud File Hosting (S3):** (Attachments will be parsed for text in-memory and discarded).
*   **Background Job Queues:** (Embeddings will be handled synchronously with strict text-length limits).
